# Case 07 — 시연 대본 (영수증 일괄 OCR → 경비 정리 엑셀)

박과장 페르소나. AX상사 관리부 박과장이 매월 모은 영수증 100장을 수기 입력하는 야근을 끝내는 시연.

## 1분 버전 (고객 미팅 도입부)

**[10초] 페인 제시**
> "AX상사 박과장님이 매월 영수증 100장 수기 입력하는데 3시간. 마감 가까워지면 야근 단골이죠. 영수증 사진 한 장 한 장 보면서 거래일·가맹점·금액 받아치는 일."

**[5초] 실행 (5장만)**
```bash
uv run python -m cases.case07_ocr_receipt_to_excel.scenario
# input/에 5장만 두면 5초 처리
```
→ 콘솔에 timer: `영수증 OCR (5장) 완료: 5.2s`

**[30초] 결과**
- Quick Look으로 expense_report.xlsx 열기
- "거래일·가맹점·카테고리·결제수단·금액 5컬럼. 회계SW 임포트 양식 그대로."

**[15초] 임팩트**
> "Gemma 4 E2B는 로컬 Ollama. 영수증이 외부 클라우드로 안 나갑니다. 100장도 1분 안에 끝나고요."

## 3분 버전 (메인 시연)
1분 버전 + 다음 추가:

**[+30초] 100장 일괄 시연**
- input/ 비우고 `personas/sample_data/receipts/` 100장 폴더 사용
- 처리 중 콘솔 출력 시연: 영수증당 600ms, 합산 ~1분
- `[OCR] r042.png → 스타벅스 강남점 / 5500원 / 2026-04-15`

**[+45초] 정확도 측정**
- `_ground_truth.json` 펼쳐서 보여주기
- 가맹점 일치율 / 금액 일치율 / 일자 일치율 — 직접 계산하는 코드는 Phase 2.5 후속

**[+15초] Q&A 핵심**
- "영수증이 회사 밖으로 나가나?" → "전부 로컬 Ollama. 외부 API 호출 0건."
- "Gemma 4 무거운데 노트북에서?" → "E2B는 4GB RAM이면 충분. M1 air에서도 영수증당 600ms."

## 5분 버전 (강의)
3분 버전 + 다음 추가:

**[+1분] 회계SW 임포트 시연**
- 결과 xlsx → 회계SW (가상) 거래입력 화면에 import
- 5컬럼이 그대로 매핑되는 화면 시연

**[+1분] 가상 데이터 한계 명시 (정직성)**
> "여기 시연한 100장은 Pillow로 합성한 가상 영수증입니다. 노이즈·회전·블러를 적용해서 OCR이 만만하게 못 맞추게 했지만, 실 영수증과 100% 동일한 환경은 아닙니다. 실제 도입 전엔 사용 환경 영수증 10장 정도 hold-out 검증을 권장합니다."

**[+30초] 향후 개선**
- E4B (8B 모델) 비교 — case08에서 진행
- 결제수단·카테고리 자동 분류 정확도 향상 (현재는 가맹점 prefix 룰베이스)
