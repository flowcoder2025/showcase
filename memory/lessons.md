- [2026-05-11] **Python import 시스템 가정은 plan note 라도 verify 필수 — `__getattr__` 자동 해결 + `sys.meta_path.append` 둘 다 실 동작과 다름** (T44, shim 안정화). plan v2.1.1 T44 line 1547 가 "Python 의 ModuleType `__getattr__` 는 import 시점에 자동 해결되므로 명시 불필요" 라고 주석 처리했지만 실 동작은 **`__getattr__` 가 attribute access hook 이라 import resolution 단계에서 호출되지 않는다**. ``from core.excel.reader import read_excel`` 같은 deep import 는 Python 의 import 시스템이 ``core.excel`` sub-module 을 별도 ``find_spec`` 으로 찾는데 attribute access 가 아님 → ``ModuleNotFoundError: No module named 'core.excel'``. 정답은 ``importlib.abc.MetaPathFinder`` + ``Loader`` 를 ``sys.meta_path`` 에 등록해 ``core.*`` import 를 가로채는 것. 추가로 ``sys.meta_path.append(...)`` (단순 등록) 도 안 됨 — 기본 ``PathFinder`` 가 우리 finder 보다 앞에 호출되어 sub-sub 가 path-based 로 별도 module 객체로 import 됨 (alias 깨짐). ``insert(0, ...)`` 로 정렬해야 alias 동작. **교훈 (3가지)**: (1) Python import internals 가정은 plan note 라도 ``__getattr__`` / ``__path__`` / ``MetaPathFinder`` ordering 같은 detail 까지 docs 또는 실 코드로 verify. "자동 해결" 표현은 의심 신호. (2) shim 같은 import-time redirect 구조는 sys.meta_path 등록 순서가 결정적 — 기본 PathFinder 앞에 insert 안 하면 sub-sub 깨짐. test 로 동일성 (`is`) 검증 필수. (3) plan code 의 example symbol (예시 `read_excel`) 도 실 export 여부 grep 확인 — 우리 라이브러리는 `read_dir` 만 export 였음. lessons.md [2026-05-01] "plan code 외부 의존성 ls/grep 검증" 의 import-time 버전.
